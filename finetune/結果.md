## ★ v2 更新（2026-06-28 後續）：字幕斷句修復（標點重訓 + aligner 修正）

### 問題
v1 辨識品質好，但**輸出幾乎無標點**（訓練 `text` 標點僅 0.16%、空白 4.16%），字幕斷句只能靠
`break_on_space` + `MAX_CHARS=20` 硬切 → 快語速長段被攔腰切在詞中間（「情勢其/實」「打左/邊」）。
**根因**：`03_make_clips.py` 把窗口內各 cue 用 `"".join` 串接，**抹掉了創作者每行字幕（=天然語句邊界）**，
模型學不到一致邊界（那 4.16% 空白全是 cue 內部的英文詞分隔 / 換行）。

### 修法 1：訓練資料補回 cue 邊界（逗號）
`03_make_clips.py`：cue 串接改為
`text = "，".join(normalize_text(c["text"]) for c in w)`（全形「，」**插在 normalize 之後**保持原樣，
插之前會被 `_fullwidth_alnum_to_half` 壓成 ASCII ","；`raw_text` 維持原 `"".join` 供溯源）。
→ 逗號密度 **0.16%→7.58%**，模型原生在語句邊界吐「，」。
**計分中性**：`for_cer` 對稱去標點，CER/entity 與 v1 可直接比較；專名不受影響（只插在 cue 邊界）。

### 修法 2：ForcedAligner 渲染漂移（`app-gpu.py:_ts_to_subtitle_lines`）
驗證時發現殘留壞斷點**不是模型、是渲染**：FA 的 `tokenize_space_lang` 會丟棄非字母數字/CJK 符號（如「%」），
舊版用 `word_list` 重建文字 + 等長消耗 `raw_text` → 遇「%」失步、逗號掛錯詞、且輸出掉字（37%→37）。
改為**以 `raw_text` 為準切行（規則同 fallback `_split_to_lines`），FA 詞級時間只依字元穩健對位後借用**
（每個 word 的字元在 raw_text 逐一對位、跳過對不上的符號，forward/backward fill 補時間）。
→ 切點正確、「%」保留、時間軸仍精準（字級）。**只影響 aligner 路徑；fallback `_split_to_lines` 不動。**

### 結果（yt5 test，full 204；計分同 §2）
| 變體 | CER | entity micro-F | recall |
|---|---|---|---|
| 1.7B+LoRA **v1**（曾部署） | 0.1993 | 0.933 | 0.882 |
| **🏆 1.7B+LoRA v2（逗號重訓，現部署）** | **0.1995** | **0.950** | **0.906** |

→ CER 持平（+0.0002）、**entity-F +1.7pt、recall +2.4pt**；斷句在 9:00~9:30 由「詞中硬切」→「句句完整」。
v2 訓練 val 觸底 epoch1（best_val 0.326，略優於 v1 ~0.337）。

### 部署（`app-gpu.py`）
- `_PREFERRED_GPU_MODEL = "pkm-ft-1.7b-v2"`（v1 保留在 `GPUModel/` 可回滾）。
- 新增 `_model_has_punct(name)` + `_PUNCTLESS_GPU_MODELS={"pkm-ft-1.7b-v1","pkm-ft-v0"}`：
  基座與 v2+ 微調帶標點 → `break_on_space` 預設**關**（靠標點斷句）；舊版 v0/v1 無標點 → 預設**開**。
  OpenCC 仍依 `_model_outputs_traditional`（v2 微調 → 關）。
- 兩檔 py_compile 通過；**待使用者開 app 實機驗證 GUI**。

## ★ v3 更新（2026-06-29）：純 YT1 訓練（刻意過擬合 YT1 講者）

### 動機與設定
需求＝**訓練集全部只用 YT1、先不管 overfitting**，最大化對 YT1 頻道（講者 Fiske）的擬合。
先把 `yt_source.txt` 的 YT1 由 6 → **15 部**（新增 9 部近兩月影片），重跑 `02→03→04`，
再用新增的 `scripts/make_yt1_subset.py`（篩 `speaker=="yt1"`）產出純 YT1 子集：
- **`manifests/train.yt1.jsonl`**：4467 clips / **12.96h**（15 部）
- **`manifests/val.yt1.jsonl`**：497 clips / 1.40h（各片尾段 10%）
- YT1 總計 **14.36h**。

配方與 v2 完全相同（r16/alpha32/dropout0.05、lr2e-4 cosine、3 epoch、eff batch 32、eval-every 40、bf16/sdpa），
僅換 `--train/--val` 指向 yt1 子集、`--out adapters/p3-1.7b-v3`。
**`best_val=0.2166`**（step280；遠低於 v2 混合資料的 0.326，反映對 YT1 in-domain 擬合很強）；
train loss ep0 0.41 → ep1 0.16 → ep2 0.11（如預期強力過擬合）。merge → **`GPUModel/pkm-ft-1.7b-v3`**。

### 結果（yt5 test，full 204，context off）
| 變體 | CER | entity micro-F | recall |
|---|---|---|---|
| 1.7B+LoRA **v2**（混合 yt1-4，現部署） | **0.1995** | **0.950** | **0.906** |
| 1.7B+LoRA **v3**（純 yt1） | 0.2171 | 0.931 | 0.898 |

→ v3 在 yt5 **略遜 v2**（CER +1.8pt、entity-F −1.9pt；TP114/FP4/FN13）。**這是預期的過擬合代價**：
**yt5 是「不同講者」hold-out**（講者 Sharlin，非 YT1 的 Fiske；樣本可見 v3 把「Sharlin」聽成「小林」），
v3 只學一位講者 → 跨講者泛化變弱。但 in-domain val loss 大幅較低 → **對 YT1 同風格內容預期更準**，
惟全 YT1 已進訓練、**無乾淨的 YT1 hold-out 可量化證明**（若要量化，需保留一部 YT1 當測試集重訓）。

### 部署決定
- **不改預設**（仍 `_PREFERRED_GPU_MODEL = "pkm-ft-1.7b-v2"`）。`pkm-ft-1.7b-v3` 已在 `GPUModel/`，
  app 模型選擇器自動掃到 → 下拉選單可手動切到 v3 實測。
- ⚠️ **資料副作用**：重跑 03/04 時 `raw/general/`（新聞，有 srt）被一併切進重生的 `train.clean.jsonl`，
  故**現在的 `train.clean.jsonl` 已含 general + 15 部 yt1，不再是 v2 當時那份**（v3 只吃 `train.yt1.jsonl`，不受影響；
  獨立的 `general.jsonl` 遺忘檢查清單未被動到）。日後若要做「混合資料」訓練，先確認 `train.clean.jsonl` 內容。

## ★ v4 更新（2026-06-29 晚）：同音/專名誤判修正（撒嬌・戲法空間・仙子伊布）

### 動機
V3 實測（`pokemon_srt/` Fiske 新片）固定誤判三詞：**撒嬌→沙雕、戲法空間→細化空間、仙子伊布→先一步**。
唯讀稽核確認**非標籤汙染**（錯誤詞從未出現在任何標籤），而是**訓練訊號太弱**：
撒嬌只 15 clip、戲法空間 2 clip、**仙子伊布 0 clip**（整個訓練集沒有，只在 yt5 測試集）。

### 作法（不改既有管線腳本、不重跑 02/03/04）
clip 實體檔已切好、audio 路徑與所屬 manifest 無關 → 只在 manifest 層重組。
1. **重組 split**（新增 `make_v4_split.py`，依 video_id）：把 yt5 的 `h51REmf4JNY` 移進訓練補滿三詞訊號；
   其餘兩部 yt5（`Jwy3M2W987c`、`VM6LcDCdV1M`）留作 hold-out，加 yt4 `NTt8wMs6OUE` 補時長 →
   - `train.v4.jsonl`：4529 clips / 13.16h（15 yt1 + h51；三詞 撒嬌16/戲法6/仙子7）
   - `val.v4.jsonl`：= val.yt1（497，**無三詞**，僅監控 val loss）
   - `test.v4.jsonl`：326 clips（2 yt5 留出 + yt4；hold-out 戲法~14/仙子~9，撒嬌 0）
2. **過取樣難詞**（新增 `make_v4_replay.py` + train_lora 內建 `--replay`）：篩三詞 clip（去重 27）各 ×8 →
   `replay.v4.jsonl`（216 列），每詞每 epoch 曝光 撒嬌~144 / 戲法~54 / 仙子~63。
3. 配方同 v2/v3，`--out adapters/p3-1.7b-v4`。**best_val=0.2190**（ep1）。

### 結果（檢查點掃描，採 `last`）
`val.v4` 不含三詞 → 不能只靠 val loss 選 best；掃 `best`/`last` 在 `test.v4` + 撒嬌實測切片逐詞比對：

| 指標 | V3 | v4 best(ep1) | **v4 last(ep2)←採用** |
|---|---|---|---|
| test.v4 CER | — | 0.108 | 0.1125 |
| test.v4 entity micro-F / recall | — | 0.915 / 0.869 | 0.926 / 0.869（FP1）|
| **戲法空間** recall（hold-out 14） | **3/14** | 8/14 | **9/14** |
| **仙子伊布** recall（hold-out 9） | **0/9** | 5/9 | **5/9** |
| **撒嬌**（實測切片） | ✓ | ✓ | ✓ |

→ **戲法空間 3→9、仙子伊布 0→5**（V3 完全不會、訓練本來 0 筆，靠移入 h51＋過取樣救起）。
採 `last`（三詞最佳、FP 最低、CER 幾乎不變）merge → **`GPUModel/pkm-ft-1.7b-v4`**。

⚠️ **撒嬌 caveat**：離線（18s 長窗、~4s 短窗）測試中 **V3 與 V4 都正確輸出「撒嬌」**，
**無法重現 production 的「沙雕」** → 該失敗應為 **app 端 VAD 切窗/解碼路徑**所致（非裸模型）。
V4 把撒嬌曝光拉到 ~144/epoch 讓先驗更強、短窗更穩，但**最終須在 app 選 V4 重轉該片確認**；
若仍出現沙雕，要查 app 切窗（與模型無關）。

### 部署決定
- **不改預設**（仍 `_PREFERRED_GPU_MODEL = "pkm-ft-1.7b-v2"`）；`pkm-ft-1.7b-v4` 已在 `GPUModel/`，
  app 下拉可手動切。**app 重轉 Fiske 片、三詞正確** 驗收通過後再考慮改預設指向 v4。
- `test.v4` 組成與 v2/v3 舊 test（3 部 yt5）不同 → **CER 不可直接跨版比**；以「同題目逐模型比對」與 hold-out 逐詞 recall 為準。

## 3. 對原計畫的 3 個重要修正（都經實測，後續務必沿用）

1. **test 用 yt5，不是 yt3**：yt3 只有單部 20.6min，過小。yt5（3 部 ~45min/204clips）達標。
   train = yt1+yt2+yt3+yt4（~8.9h、4 講者），val = 各 train 影片時間尾段 10%。

2. **不要做 OpenCC 繁簡/詞彙轉換**（最重要）：來源字幕本就是乾淨繁體。實測 OpenCC **破壞專名**：
   `岩崩→巖崩`、`仆斬將軍→僕斬將軍`、`洛托姆→洛託姆`、`類型→型別`（s2twp 詞表誤觸）；
   連字元級 `s2t` 也把通用字誤判（岩/仆/只/吃/才/系/群/托→冷僻異體）。真簡體字 <0.3% 且多假陽性。
   → 訓練目標＝創作者字幕**逐字**。文字校正只靠 canonical lexicon 對「已知專名錯拼」保守替換。
   → `for_cer` 的 t2s 只在**計分**時對 ref/hyp **對稱**套用（消除繁簡軸測內容），與輸出無關。

3. **生產 `app-gpu.py:598`/`:690` 的 OpenCC 同樣會破壞專名**：預設 `s2t`（詞彙開則 `s2twp`）。
   微調模型原生輸出正確繁體，這步反成有害 → **部署時務必關掉 OpenCC**（這就是待做 UI 開關的動機）。

其他經實測的結論：
- **偏置（context）反傷微調後的模型**：0.6B+LoRA 加偏置 → FP21、CER 變差。模型已內化專名 → **部署 context 留空**。
- yt-dlp **用原生 CLI**（自動走 android_vr，不需 cookie）；403 是間歇性，重試即可。用 Python API 逐項指定 player_client 反而搞壞 client 選擇。

## 4. 已驗證的技術接線（計畫 6 條 BLOCKER/ERROR 全部對真實 0.6B 驗證通過）

`scripts/verify_harness.py` 一次驗證以下全綠（保留可重跑）：

- **BLOCKER-1**：loss 走 `model.thinker(input_ids, input_features, attention_mask, feature_attention_mask, labels).loss`（top-level `Qwen3ASRForConditionalGeneration` 只有 `generate()`）。
- **BLOCKER-2**：LoRA target_modules 精確 regex 只掛 thinker LM：
  `r"thinker\.model\.layers\.\d+\.(self_attn\.(q|k|v|o)_proj|mlp\.(gate|up|down)_proj)"`
  → 實測命中 thinker=196、**audio_tower=0**。
- **ERROR-3**：audio_pad 數量**動態**（processor 依音訊長度產生）：10s→130、7s→91，**不是寫死 390**（390 只對 30s）。
- **ERROR-4**：手建 labels（`scripts/asr_labels.py`）。序列 =
  `prefix + N×audio_pad(151676) + suffix[151670,151645,198,151644,77091,198] + zh語尾[11528,8453,151704] + 轉錄token + eos(151645)`；
  **只監督「語尾+轉錄+eos」其餘 -100**；斷言 `labels.shape==input_ids` 且 `(labels!=-100).sum()==尾巴token數`。
  collator padding：labels 補 -100、長度對齊。
- **ERROR-5**：`scripts/merge_lora.py` merge_and_unload → **複製原始目錄** → 刪舊 `model*.safetensors`+index →
  `save_pretrained`（保留 config/preprocessor/chat_template/tokenizer）→ `qwen_asr.Qwen3ASRModel.from_pretrained` 載入 → transcribe → `parse_asr_output` 無誤。
- **GOTCHA（計畫沒寫）**：`gradient_checkpointing_enable()` / `enable_input_require_grads()` 必須對 **`model.thinker`** 呼叫，
  不能對 top-level（top-level 未實作 `get_input_embeddings`，會 NotImplementedError）。
- **GOTCHA**：merge 後 `save_pretrained` 會被 transformers 4.57 嚴格驗證擋下（`do_sample=False` 不可有 `temperature`）→
  存檔前 sanitize `generation_config`（`temperature/top_p/top_k=None`）。

過擬合煙霧：24 clips loss 7→**0.0003**（73s）。整個訓練/merge/載入鏈路確定正確。
